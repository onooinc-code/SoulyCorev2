HedraCore Feature Demonstrator

Style & Content Manual (v3)

1. Core Philosophy

Single-User Centric: The primary directive is to build a professional tool for a single power user. All features must serve the goal of enhancing personal productivity, control, and workflow automation. Features related to community, marketplaces, social interaction, team collaboration, or public sharing are explicitly excluded.

API-First Architecture: HedraCore is the "Agents Memory and Tasks Manage (Brains)" engine within a larger system. The UI is a primary client, but not the only one. Every feature must be designed as a UI layer over a robust, well-defined backend API. Descriptions should reflect this by referencing endpoints (e.g., /api/v2/context/inject), data models, and backend processes.

2. Layout & Structure

  • Full-Width Layout: The main content area should utilize the full width of the viewport, removing any horizontal `max-width` constraints. This provides an expansive, professional feel suitable for a power tool. The header may remain centered.
  • Three-Part Content Flow: Each demonstrator file must follow this exact structure:
    1. An **All-English Feature Table** at the top.
    2. A second **Mixed-Language Feature Table** (English names, Arabic details).
    3. A final section of **Arabic Expandable Panels** with detailed explanations and UI mockups.

3. Content Guidelines

  • Feature Count: Each demonstrator file must contain between 50 and 100 unique features, not including the "Crazy Ideas" section.
  • Bilingual Structure:
    • English Table: All columns (Feature Name, Category, Description, etc.) must be in English.
    • Arabic Table: The 'Feature Name' column must remain in English for easy cross-referencing. All other columns must be in Egyptian Arabic.
    • Arabic Explanation Panels: The title, detailed explanation, and example within these panels must be in Egyptian Arabic.
    • Technical Terms: Do not translate technical and programming terms. Words like "API", "JSON", "endpoint", "backend", "frontend", "schema", "sandbox", "CLI", "SDK", "database", "cache" must remain in English for clarity.
  • UI/UX Mockups: Each Arabic explanation panel MUST contain a simple, visual mockup of the UI feature being described. This can be achieved using basic HTML elements and Tailwind CSS classes to represent buttons, inputs, messages, etc. This is critical for demonstrating the concept visually.
  • "Crazy Ideas" Section: This section is mandatory. It should showcase forward-thinking, speculative concepts. The format should be cards or panels with explanations in Egyptian Arabic.

4. Functionality

  • Functional Export: The "Export Selected" button must be fully functional.
  • Export Content: When clicked, the script should gather the following for each selected feature:
    • English Name, Category, and Description.
    • Arabic Category and Description.
    • A plausible, sample **Business Goal**.
    • A plausible, sample **Technical Specification** (e.g., API endpoint, request body, response).
  • Export Format: The gathered information should be formatted into a single, structured string optimized for an AI agent to parse.
  • Export Action: The final string must be copied to the user's clipboard using navigator.clipboard.writeText().
  • User Feedback: Provide clear feedback to the user, such as an alert or changing the button text to "Copied!".
